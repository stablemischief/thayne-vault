## Nesvick Open Notebook — Critical Auth Architecture Problem

### Root Cause Identified (Mar 17 morning)
Archon's holistic analysis ran but produced a plan.md for the wrong task (transcript download from Stable-Notebook history). The ACTUAL root cause of the broken app was identified manually:

**In `api/main.py`:**
```python
_jwt_dep = [Depends(get_current_user)]
```
Applied to EVERY data router (notebooks, settings, credentials, etc.) — requires valid Clerk JWT Bearer token on every API call.

**The contradiction:**
- `ClerkTokenProvider` was calling `getToken()` from Clerk's `useAuth()` hook
- `getToken()` hangs indefinitely client-side after OAuth sign-in
- We removed `ClerkTokenProvider` (PR #11) to unblock the app
- But without it, no JWT is sent to FastAPI → 401 on every data call
- React Query retries in a loop → rate limiter triggers → 429 → "Unable to Connect"

### Architecture Decision Needed
The correct long-term approach for Nesvick:
- Option A: Remove `_jwt_dep` from data routes (Clerk protects at Next.js level, API is internal-only)
- Option B: Proxy all browser API calls through Next.js server routes (adds server-side JWT)
Option A is simpler and correct for single-tenant/Docker deployment. Option B is more secure for multi-tenant.
