## 🚨 GLOBAL RULE — Issue Brief Standard Before Archon (NON-NEGOTIABLE)

**This process applies to EVERY issue, EVERY repo, EVERY channel, NO exceptions.**

Before dispatching Archon on ANY issue, the GitHub issue MUST contain ALL of the following:

1. **Problem Statement** — what fails/is missing, observed vs expected, exact errors, file paths, function names
2. **Proposed Solution** — exact files to change, before/after text, function signatures, patterns to follow
3. **Code Examples** — concrete before/after snippets, input/output samples, type definitions, JSX/Python as applicable
4. **Approved Mockup (for UI issues)** — the FULL HTML of the approved mockup embedded in the issue body as a code block. Not a file path. Not a reference. The actual HTML. Archon must be able to read it directly.
5. **Key Files to Reference** — every file Archon should read before touching anything
6. **Acceptance Criteria** — numbered checklist, each item verifiable, tests pass before AND after
7. **What NOT to Change** — explicit list of files and behaviors that must remain untouched
8. **Research Findings** — facts Thane observed: code snippets, line numbers, what was ruled out

**Step order (NON-NEGOTIABLE):**
1. Write issue with Problem Statement + Proposed Solution + Key Files
2. Create Excalidraw user flow diagram (if applicable)
3. Build 3 HTML/CSS mockups — James selects one
4. Update issue with selected mockup HTML embedded + Code Examples + Acceptance Criteria + What NOT to Change + Research Findings
5. ONLY THEN dispatch Archon
